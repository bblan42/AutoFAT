HMI Testing
Feature in design — input needed before build
Planned Feature — Not Yet Built

HMI testing is a distinct verification layer from I/O and comms testing. The current FAT documents include an HMI Expected column in each test table, but there is no formal structure for testing HMI displays, faceplates, alarm pages, trend historians, or navigation paths as standalone test cases.

This section will cover structured checklist-based verification of the operator interface — independent from the underlying signal verification. A technician can confirm the field signal while the client representative simultaneously verifies the HMI display, with both signing the same document.

Proposed Test Structure

Each HMI test case would produce a checklist document with the following format:

Screen / Faceplate Navigation Path Element Expected Display Actual P / F
FT-101 Faceplate Overview → Reactor → FT-101 EU Readout 0 – 500 GPM
FT-101 Faceplate HH Alarm Badge Red, latches on activation
Reactor Overview Main → Reactor Unit Tag label position Upper-left of heat exchanger graphic
Open Questions — Needed Before Build
Q1 — Bundled or Standalone?
Should HMI checks be a standalone test type (separate document, separate sign-off), or an optional section bundled into each AI/DI/Motor test — so the field tech and client rep verify signal and display on one page?
Q2 — HMI Platform
Which HMI platforms need to be covered? WinCC, FactoryTalk View, Ignition, iFIX, Wonderware/AVEVA, other? Does the platform affect the test structure, or is it just metadata on the document header?
Q3 — Screenshot Attachment
Should the generated document include placeholder boxes for screenshot attachments (paste in a screen capture after the FAT), or is a hand-written description in the Actual column sufficient for your clients?
Q4 — Alarm Page Testing
Do you need a specific sub-type for verifying the alarm summary page (alarm appears, priority is correct, message text matches datasheet)? Or is that covered by the alarm verification test + the HMI faceplate check?
Q5 — Navigation / Workflow Tests
Do you need a test type for overall navigation (can the operator reach every screen within N clicks? are emergency shutdown buttons accessible from all screens?) separate from per-tag display tests?
