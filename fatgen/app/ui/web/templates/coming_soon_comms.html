Communications Testing
Modbus / Profinet / OPC-UA — Feature in design
Planned Feature — Not Yet Built

Signal-level scaling tests verify the 4–20 mA loop. Comms tests verify that the same value appears correctly at every system boundary above the PLC: SCADA, historian, DCS gateway, third-party safety system, or remote I/O. This is a separate, often overlooked failure mode — the PLC can read correctly while Modbus register scaling or OPC tag mapping is wrong.

Each protocol has a different test pattern. The documents generated will follow the same structure as scaling tests — inject a known value at the PLC, read it back through the comms interface, verify the result matches expected scaling.

Protocol Coverage Planned
MODBUS TCP / RTU
  • Register address mapping (40001–4xxxx)
  • Data type verification (INT16, UINT16, FLOAT32)
  • Read / write / read-back test cycle
  • Coil (0x / 1x) discrete verification
  • Byte order (Big/Little endian)
  • Scaling factor / offset check
PROFINET
  • Device slot/subslot mapping
  • Input/output data consistency
  • Device status / AR state
  • I&M data (vendor, order number)
  • Alarm reporting (process alarm, pull/plug)
  • MRP ring redundancy (if applicable)
OPC-UA
  • Node read by NodeId / Browse path
  • Data type match (Float, Int32, Bool)
  • Write + read-back cycle
  • Subscription / change notification
  • Security mode (None / Sign / SignEncrypt)
  • Historical data access (if historian present)

Open Questions — Needed Before Build
Q1 — Test Boundary
Are you testing at the PLC gateway output (a laptop running Modbus poll software reads the PLC directly), or end-to-end (SCADA historian tag reads the PLC, you verify the historian value)? This changes whether we need to track a SCADA tag name alongside the PLC address.
Q2 — Protocol per Project or per Tag
Is the comms protocol a project-level setting (one protocol per FAT), or do individual tags route to different destinations (e.g., flows go via Modbus to the DCS, safety trips go via OPC-UA to the SIS)?
Q3 — Register Map Source
Is the Modbus register map / OPC-UA node list something the engineer enters manually, or is there an export format from the PLC IDE or gateway config tool that we could import? (e.g., a CSV from Kepware, Ignition, or WinCC that maps tags to register addresses)
Q4 — Bi-directional (Write) Testing
For AO and DO tags — do you need to test writes from the SCADA/DCS down to the PLC (e.g., SCADA writes a setpoint via Modbus, PLC acts on it)? Or is comms testing read-only (verify the PLC value appears correctly at the SCADA)?
Q5 — Profinet Priority
Profinet testing is more complex (requires a network analyzer or diagnostic tool). Is this a priority for Phase 1 comms, or should Modbus TCP and OPC-UA come first?
